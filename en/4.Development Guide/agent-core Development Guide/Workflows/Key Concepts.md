# Workflow Components

Workflow components are the fundamental functional units that constitute a workflow, each undertaking specific business logic or data processing tasks. Through data interaction and collaboration between components, they support asynchronous processing, streaming data transmission, and dynamic orchestration, thereby enabling efficient and flexible workflow execution.

The following sections introduce components in terms of classification, capability types, and input/output.

## Component Classification

Components are divided into built-in components and custom components.

- **Built-in Components**: openJiuwen includes two categories of components to help users quickly build workflows:
  
  - **Basic Components**: Used for flow control, including Start components, End components, Branch components, Loop components, Sub-workflow components, etc., responsible for defining and managing the execution logic of the workflow.
  - **Functional Components**: Used to provide business function support, including LLM components, Intent Recognition components, Questioner components, Plugin components, etc. They are out-of-the-box and particularly suitable for interacting with LLMs in Agent workflows.
- **Custom Components**: Based on the component extension capabilities provided by openJiuwen, developers can conveniently develop custom components.

## Component Capability Types

In workflow components, `invoke`, `stream`, `transform`, and `collect` are four core capabilities, catering to different needs of batch processing and stream processing respectively:

* `invoke`: Corresponds to the enum value `ComponentAbility.INVOKE`. It indicates that batch data is processed all at once and output as batch data. It is suitable for scenarios requiring holistic data processing and has a wide range of applications.
* `stream`: Corresponds to the enum value `ComponentAbility.STREAM`. It indicates converting batch data into stream data, requiring item-by-item processing. It is suitable for scenarios requiring real-time or step-by-step data processing, such as generating streaming responses from LLM batch requests or analyzing data item by item.
* `collect`: Corresponds to the enum value `ComponentAbility.COLLECT`. It indicates collecting stream data into batch data. It is suitable for scenarios requiring the summarization, aggregation, or final result generation of streaming output, such as collecting and organizing the streaming output of an LLM before returning the final result.
* `transform`: Corresponds to the enum value `ComponentAbility.TRANSFORM`. It indicates performing format conversion or content processing on stream data and outputting it as stream data. It is suitable for scenarios requiring frame-by-frame or item-by-item processing, format conversion, or content enhancement of streaming data, such as formatting the streaming output of an LLM frame by frame before outputting.

For specific implementation details of component capabilities, please refer to [Capability types supported by the component](./Using%20Components/Capability%20types%20supported%20by%20the%20component.md).

## Component Input and Output

Components support input and output in two forms: streaming and batch data. In the data processing chain, raw data is formatted at the component's input end for internal use. At the component's output end, the processing results are formatted and distributed to downstream components. For details, please refer to [Defining component input and output formats](./Using%20Components/Define%20component%20input%20and%20output%20formats.md).

This out-of-the-box built-in component library facilitates users in quickly building workflows and improving workflow construction efficiency, while also providing component extension capabilities to allow users to enrich the functional ecosystem of the entire application.

# Connections

The connection relationship between components describes the flow path of data, control signals, or messages between components. Connections are divided into three types: General Connection, Conditional Connection, and Streaming Connection.

* **General Connection**: The most basic form of component connection, used to describe the sequential dependency between components, meaning the execution of a component depends on the output of its predecessor component.
* **Conditional Connection**: Introduces conditional judgment logic within the connection, used to decide whether a specific component or branch should be executed. For example, an Intent Recognition component can choose to enter different branch paths based on different recognition results, thereby achieving dynamic branch control.
* **Streaming Connection**: Used to express the streaming data transmission relationship between components. The components at both ends are typically referred to as the Producer and the Consumer. Streaming data generated by the Producer component is transmitted in real-time to the Consumer component through the streaming connection, thereby supporting real-time stream processing. Streaming connections are widely used in scenarios such as LLM output Token data streams.

This diverse connection mechanism enables workflows to balance the needs for sequential execution, conditional control, and real-time processing, enhancing the flexibility and expressive power of business processes.

# Workflow Session Propagation

The workflow session (Session) is the core carrier of workflow execution, responsible for passing necessary data and state information between components to ensure the consistency and integrity of the entire workflow execution process.

Each time a workflow is invoked, the caller needs to pass in a workflow session (usually created by `create_workflow_session()`). This session encapsulates multiple categories of information required for workflow execution:

* **Workflow Configuration**: Defines the overall running parameters of the workflow.
* **Component Configuration**: Records the personalized parameters of each component.
* **Component State**: Saves the intermediate states of components during operation.
* **Global State**: Supports data sharing between different components.

During operation, components can read and write data through the session, thereby achieving collaboration with other components. In addition, the session also provides access capabilities to framework functional modules, such as streaming output, further extending the capability boundaries of components.

Through this design, the session serves not only as a centralized management center for data and state but also as the infrastructure for collaboration and extension between components.

# Workflow Composition

Different workflows can achieve nested calls through workflow components, where the called workflow is referred to as a Sub-Workflow. This method of workflow composition not only breaks the boundaries of a single workflow but also provides stronger expressive power for complex business scenarios.

Through the sub-workflow mechanism, users can:

* **Decompose Complex Tasks**: Break down a large process into multiple independent sub-workflows, where each sub-workflow focuses on handling specific sub-tasks, resulting in a clearer structure.
* **Improve Reusability**: Common business logic can be abstracted into sub-workflows and called multiple times in different main workflows, avoiding duplicate implementation.
* **Enhance Flexibility**: The main workflow can dynamically call different sub-workflows based on runtime conditions, thereby achieving flexible process orchestration.
* **Facilitate Maintenance and Extension**: As relatively independent modules, sub-workflows can be developed, tested, and iterated separately, reducing maintenance costs.

This composition capability based on sub-workflows enables openJiuwen workflows to not only handle conventional linear business scenarios but also gracefully support large-scale, hierarchical, and dynamic task orchestration requirements.
