# 调测能力

调测与可观测能力主要由 `Tracer` 与流式 schema 体系提供（见 `openjiuwen.core.session.tracer` 与 `openjiuwen.core.session.stream`）。

## Trace（追踪）

- 通过 `Session.trace(...)` / `Session.trace_error(...)` 记录执行过程与异常信息。
- `openjiuwen.core.session.tracer` 提供 tracer 实现与装饰器（具体接口以源码为准）。

### 参数说明：trace

```python
await session.trace(data)
```

- **data** (dict)：要记录的追踪数据。
  - 建议保持结构化，便于上报、检索与聚合。
  - 建议包含“事件名 + 阶段 + 关键字段”，例如：
    - `event`：事件名（如 `llm_call`、`tool_call`、`retrieval`、`route`）
    - `stage`：阶段（如 `start`/`end`）
    - `duration_ms`：耗时（可选，end 阶段常用）
    - `component_id`：组件/节点 id（可选，一般可从 session.executable_id() 推导）
    - `metadata`：任意补充信息（例如 token 用量、命中率、top_k 等）
  - 避免写入敏感信息（密钥、用户隐私、原始长文本等）；可写入摘要或引用 id。

### 参数说明：trace_error

```python
await session.trace_error(error)
```

- **error** (Exception)：异常对象。
  - 建议在捕获异常后尽快调用，以便 tracer 记录堆栈与上下文。
  - 若你会在上层重新包装异常，建议先 `trace_error` 再 raise，避免丢失原始错误信息。

## 流式调试输出

- 组件可以通过 `session.write_stream(...)` 输出结构化帧（例如步骤、事件、指标等）。
- 上层可通过 stream iterator 消费并呈现实时调试信息。

### 推荐做法

- **把“调试信息”也结构化**：例如用 `type="step"` / `type="metric"` / `type="event"`。
- **把关键上下文放进 payload**：例如本次调用的输入摘要、输出摘要、耗时、分支选择结果。
- **保证 payload 稳定**：避免同一 type 在不同组件输出完全不同结构，降低前端/消费端的解析复杂度。
