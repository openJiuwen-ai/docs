# 上下文引擎

上下文引擎（`ContextEngine`）是 openJiuwen 在 0.1.4 中推荐的统一上下文管理组件，主要负责**模型输入上下文**的构建与复用。它围绕 `Session` 和 `ModelContext` 工作，帮助你在复杂工作流 / Agent 执行过程中，稳定地管理：

- 历史对话消息（短期记忆、窗口裁剪）
- 与记忆引擎打通后的长期记忆
- Token 统计与窗口控制
- 多个「上下文 ID」并存的场景（例如一个会话里同时跑多条子任务）

> 旧版基于 `ReActAgent.context_engine.get_agent_context/get_workflow_context` 的接口已不再是推荐用法，本章直接以 `ContextEngine + Session + ModelContext` 为核心展开。

## 核心概念

- **ContextEngine**：
  - 入口类：`openjiuwen.core.context_engine.ContextEngine`。
  - 负责按 `session_id + context_id` 管理多个 `ModelContext` 实例。
  - 提供统一的 `create_context / get_context / clear_context / save_contexts` 能力。
- **ModelContext**：
  - 代表**某个会话中的一个具体“对话上下文”**。
  - 负责维护消息列表、构造窗口（`get_context_window()`）、与记忆引擎交互等。
- **Session**：
  - 承载执行期状态、配置与流式输出等能力（见 `Session` 章节）。
  - `ContextEngine` 通过 `Session` 获取会话 ID、与记忆引擎协作。
- **mem_scope_id**：
  - 记忆作用域 ID，用于在同一会话下区分不同记忆空间（例如按业务线/任务划分）。
  - 和记忆引擎配合时，用于从长期记忆中加载/写入对应范围的对话片段。

## 创建 ContextEngine 实例

在应用初始化阶段（例如 Runner / Web 服务启动时），通常会创建一个全局的 `ContextEngine` 实例：

```python
from openjiuwen.core.context_engine import ContextEngine, ContextEngineConfig

context_engine = ContextEngine(config=ContextEngineConfig())
```

**配置项（示意）：**

- 默认窗口消息数：控制 `ModelContext` 在构造窗口时最多包含多少条历史消息。
- 从记忆中恢复的消息条数：控制从记忆引擎拉取多少条历史。
- 其他与裁剪策略、角色过滤等相关的参数（以源码 `ContextEngineConfig` 定义为准）。

> 一般建议在应用内**复用同一个 ContextEngine 实例**，而不是每次调用都新建一个。

## 在 Workflow 中使用上下文

在工作流组件中，推荐通过 `Session` + 全局 `ContextEngine` 管理上下文。

### 典型调用流程

1. Runner / 应用层创建并持有 `context_engine` 实例。
2. 执行工作流时，创建 `WorkflowSession`：
   - 例如 `session = create_workflow_session(session_id="xxx")`。
3. 在组件的 `invoke` 中，通过 `context_engine.create_context(...)` 获取 `ModelContext`。
4. 使用 `ModelContext` 追加消息、构造模型输入窗口，并在需要时调用 `save_contexts` 持久化。

### 代码示例：组件中使用 ModelContext

```python
from openjiuwen.core.workflow import WorkflowComponent
from openjiuwen.core.context_engine import ContextEngine
from openjiuwen.core.context_engine import ModelContext  # 类型引用，实际由 ContextEngine 返回
from openjiuwen.core.session.node import Session
from openjiuwen.core.graph.executable import Input, Output
from openjiuwen.core.foundation.llm.messages import HumanMessage


class ChatNode(WorkflowComponent):
    def __init__(self, context_engine: ContextEngine, context_id: str = "chat"):
        super().__init__()
        self._context_engine = context_engine
        self._context_id = context_id

    async def invoke(self, inputs: Input, session: Session, context: ModelContext) -> Output:
        user_query = inputs.get("query", "")

        # 1. 为当前 Session + context_id 创建/获取 ModelContext
        model_ctx = await self._context_engine.create_context(
            context_id=self._context_id,
            session=session,
            mem_scope_id="weather_scope",  # 可选：与记忆引擎协作
        )

        # 2. 追加本轮用户消息
        await model_ctx.add_messages(HumanMessage(content=user_query))

        # 3. 构造用于调用大模型的窗口
        window = await model_ctx.get_context_window()
        messages = window.get_messages()

        # 此处省略：调用 Model / Runner，基于 messages 完成推理
        # llm_response = await model.ainvoke(messages=messages, ...)

        # 4. 可选：保存上下文（写入记忆引擎）
        await self._context_engine.save_contexts(
            [self._context_id],
            session=session,
            mem_scope_id="weather_scope",
        )

        return {"answer": "..."}
```

**要点说明：**

- 通过 `context_id` 可以在同一 Session 下区分多个独立对话（例如 `"chat"`, `"tool_debug"` 等）。
- 通过 `mem_scope_id` 可以将上下文与某个记忆空间绑定，便于后续恢复。
- `ModelContext.get_context_window()` 返回的是「已裁剪」后的窗口，内部已考虑 token 限制与历史长度等策略。

## 在 Agent 场景中复用 ContextEngine

如果你在上层自定义了基于 `Session` 的 Agent（而非 legacy 单 Agent），也可以直接复用同一套 `ContextEngine`：

- 在 Agent 初始化时注入全局 `ContextEngine`。
- 在 `invoke/stream` 中，按 `session_id + context_id` 获取对应 `ModelContext`。
- 对话历史跨多轮调用自动在 `ModelContext` 中累积，并按需写入记忆引擎。

这种模式下，**ContextEngine 不再依赖 Agent/Workflow 的具体实现**，而是纯粹通过 `Session` 进行会话隔离，更容易在多种执行入口（Workflow / Task / AgentGroup）之间复用。

## 与记忆引擎的协作

`ContextEngine` 与记忆引擎（Memory Engine）通常以以下方式协作：

- **从长期记忆恢复上下文**：
  - 在 `create_context(...)` 时传入 `mem_scope_id`，且不显式提供 `history_messages`。
  - ContextEngine 会使用 `mem_scope_id` 从记忆引擎拉取最近一段历史消息，作为上下文起点。
- **将新消息写入长期记忆**：
  - 在对话轮结束时调用 `save_contexts(context_ids=[...], session=session, mem_scope_id=...)`。
  - 由 ContextEngine 负责把本轮新增的消息写入对应记忆空间。

> 具体「如何根据 mem_scope_id 组织存储」由记忆引擎实现决定，ContextEngine 只负责协调读写。

## 清理与生命周期管理

在长时间运行的服务中，需要考虑上下文的生命周期管理：

- 使用 `clear_context(session_id=...)`：清理某个会话下所有上下文（例如用户主动结束会话时）。
- 使用 `clear_context(session_id=..., context_id=...)`：只清理某个子对话。
- 在 Runner / 服务关闭前，可以选择调用 `save_contexts` + `clear_context`，将必要的信息落盘后释放内存。

示例：

```python
# 用户结束会话时
context_engine.clear_context(session_id="s1")

# 仅结束某个子对话
context_engine.clear_context(session_id="s1", context_id="chat")
```

## 小结

- 0.1.4 中，**上下文引擎的推荐使用方式**是：`ContextEngine(config) + Session + ModelContext`。
- 通过 `create_context/get_context/save_contexts/clear_context`，可以在不关心底层实现细节的情况下，完成：
  - 多会话隔离
  - 多上下文 ID 管理
  - 与记忆引擎的读写协作
  - Token 级别的窗口控制
- 旧的 `get_agent_context/get_workflow_context` 只在 legacy 单 Agent 里保留兼容，本章文档已完全围绕**新接口**展开。**
