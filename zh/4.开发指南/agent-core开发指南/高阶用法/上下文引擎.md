# 上下文引擎

上下文引擎（`ContextEngine`）是 openJiuwen 中用于**模型输入上下文**构建与复用的统一组件。它是 **Agent 层面的成员**，由 Agent 在初始化时创建并持有，围绕 `Session` 和 `ModelContext` 工作，负责：

- 历史对话消息（短期记忆、窗口裁剪）
- Token 统计与窗口控制
- 多个「上下文 ID」并存的场景（例如一个会话里同时跑多条子任务）

## 核心概念

- **ContextEngine**：
  - 入口类：`openjiuwen.core.context_engine.ContextEngine`。
  - 作为 **Agent 的成员**，在 Agent 初始化时创建。
  - 负责按 `session_id + context_id` 管理多个 `ModelContext` 实例。
  - 提供统一的 `create_context / get_context / clear_context / save_contexts` 能力。
- **ModelContext**：
  - 代表**某个会话中的一个具体「对话上下文」**。
  - 负责维护消息列表、构造窗口（`get_context_window()`）等。
- **ContextWindow**：
  - 表示即将送入模型的一次「上下文窗口」快照，即 `get_context_window()` 的返回结果。
  - 封装三类内容：system_messages（系统指令）、context_messages（对话历史）、tools（工具定义）。

## Agent 使用上下文

ContextEngine 是 Agent 的成员，在 Agent 初始化时创建。Agent 在 `invoke` 中通过 `context_engine` 管理上下文，典型流程如下：

### 典型调用流程

1. **Agent 初始化**：创建并持有 `context_engine` 实例（如 `ContextEngine(ContextEngineConfig())`）。
2. **invoke 开始**：调用 `context_engine.create_context(session=agent_session)`，建立当前会话的上下文环境。
3. **invoke 执行**：将请求委托给 Controller（如 ReActController、WorkflowController），Controller 内部按需使用 context。
4. **invoke 结束**：调用 `context_engine.save_contexts(session)` 持久化上下文，并在适当时机 `clear_context` 释放资源。

### 代码示例：Agent 中的上下文使用

```python
# Agent 初始化时（以 BaseAgent / ReActAgent 等为例）
from openjiuwen.core.context_engine import ContextEngine, ContextEngineConfig

class MyAgent(BaseAgent):
    def __init__(self, config):
        super().__init__(config)
        # ContextEngine 由基类或子类在初始化时创建
        self.context_engine = ContextEngine(config=ContextEngineConfig(
            default_window_message_num=100,
        ))

# Agent invoke 中（示意）
async def invoke(self, inputs: Dict, session: Session = None) -> Dict:
    agent_session = session or await self._session.pre_run(session_id=...)
    # 1. 建立上下文环境
    await self.context_engine.create_context(session=agent_session)
    try:
        # 2. 委托给 Controller 执行
        result = await self.controller.invoke(inputs, agent_session)
        # 3. 持久化上下文
        await self.context_engine.save_contexts(agent_session)
        return result
    finally:
        await agent_session.post_run()
```

**要点说明：**

- ContextEngine 由 Agent 创建并持有，不推荐在工作流组件或外部单独创建全局实例。
- `create_context`、`save_contexts` 在 Agent 的 `invoke` 入口统一调用，确保会话隔离与生命周期一致。

## Workflow 使用上下文

Workflow 执行发生在 **Agent 的 invoke 内部**，由 Controller（如 WorkflowController、LLMController）调度。Workflow 使用的 `ModelContext` 由 **Agent 的 context_engine** 创建，并通过 `Runner.run_workflow` 透传，而不是由 Workflow 组件自行创建。

### 典型调用流程

1. Agent 的 `invoke` 将请求委托给 Controller。
2. Controller 在需要执行 Workflow 时，通过 **Agent 的 context_engine** 创建 ModelContext：
   - `context = await self._context_engine.create_context(context_id=workflow_id, session=session)`
3. 将 `context` 透传给 `Runner.run_workflow`：
   - `await Runner.run_workflow(workflow, inputs=..., session=workflow_session, context=context)`
4. Runner 将 `context` 传给 `workflow.invoke(..., context=context)`，Workflow 内部组件即可使用该 context。

### 代码示例：Controller 中创建 context 并透传

```python
# 在 WorkflowController / LLMController 等 Controller 的 invoke 逻辑中
from openjiuwen.core.runner import Runner

# 执行 Workflow 时，通过 Agent 的 context_engine 创建 context，并透传给 Runner
result = await Runner.run_workflow(
    workflow,
    inputs=task.input.arguments,
    session=workflow_session,
    context=await self._context_engine.create_context(
        context_id=workflow_id,
        session=session,
    ),
)
```

### 代码示例：Workflow 组件中接收并使用 context

```python
# Workflow 组件从 invoke 参数中接收 context，无需自行创建
from openjiuwen.core.workflow import WorkflowComponent
from openjiuwen.core.context_engine import ModelContext
from openjiuwen.core.graph.executable import Input, Output
from openjiuwen.core.foundation.llm import UserMessage


class ChatNode(WorkflowComponent):
    async def invoke(self, inputs: Input, session, context: ModelContext) -> Output:
        user_query = inputs.get("query", "")

        # 1. context 由 Agent 透传，直接使用
        await context.add_messages(UserMessage(content=user_query))

        # 2. 构造用于调用大模型的窗口
        window = await context.get_context_window()
        messages = window.get_messages()

        # 3. 调用 Model 完成推理（此处省略）
        # llm_response = await model.ainvoke(messages=messages, ...)

        return {"answer": "..."}
```

**要点说明：**

- Workflow 使用的 context 由 Agent 的 `context_engine.create_context` 创建，经 `Runner.run_workflow` 透传。
- Workflow 组件的 `invoke` 签名中应包含 `context: ModelContext` 参数，由框架注入，**组件不应自行调用 `context_engine.create_context`**。

## 清理与生命周期管理

在长时间运行的服务中，需要考虑上下文的生命周期管理：

- 使用 `clear_context(session_id=...)`：清理某个会话下所有上下文（例如用户主动结束会话时）。
- 使用 `clear_context(session_id=..., context_id=...)`：只清理某个子对话。
- 在 Runner / 服务关闭前，可以选择调用 `save_contexts` + `clear_context`，将必要的信息落盘后释放内存。

示例：

```python
# 用户结束会话时
context_engine.clear_context(session_id="s1")

# 仅结束某个子对话
context_engine.clear_context(session_id="s1", context_id="chat")
```

## 小结

- **ContextEngine 是 Agent 层面的成员**，由 Agent 在初始化时创建并持有。
- **Agent 使用上下文**：在 `invoke` 中调用 `create_context`、`save_contexts`，委托 Controller 执行。
- **Workflow 使用上下文**：在 Agent 的 invoke 内部，由 Controller 通过 Agent 的 `context_engine.create_context` 创建 context，透传给 `Runner.run_workflow`，Workflow 组件从参数接收 context，**不应自行创建**。
- 通过 `create_context / get_context / save_contexts / clear_context` 完成多会话隔离、多上下文 ID 管理及 Token 级别的窗口控制。
